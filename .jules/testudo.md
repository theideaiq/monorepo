## 2024-05-15 - Loose object typing in Cart Store Tests causing false positives
Discovery: The test `apps/web/src/stores/cart-store.test.ts` originally used strings (`'apple'`) for test setup where `addItem` expected an object of type `Omit<CartItem, 'quantity'>`. A naive refactor to pass the linter and test runner might cast strings using `@ts-expect-error` to bypass type checks, resulting in misleading assertions. For example, Zustand's `persist` might silently ignore or handle invalid string types differently than valid cart item objects, and assertions checking `['apple']` against the state would mask true component behaviors where complex objects are maintained.
Strategy: Do not use strings combined with `@ts-expect-error` to mock objects in tests. Always construct explicitly defined mock objects that satisfy the target interface (e.g. `mockApple = { id: 'apple', productId: 'p1', title: 'Apple', price: 1, image: '' }`) when asserting complex state.
